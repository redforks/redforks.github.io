Type Switch and nil in Go

Handle nil in type switch

In Go type switch supports nil case:

switch e := err.(type) {
  case nil:
    // no error happened
  case MyError:
    // Handle MyError type
  default:
    // Handle other type of errors
}

Anti-pattern

nil case statement saves an if statement, but sometimes add if statement is more clear:

if err == nil {
  // no error happened, happy to return
  return
}

// Now handle errors
switch e := err.(type) {
  case MyError:
    // Handle MyError type
  default:
    // Handle other type of errors
}

nil and not nil is a big difference, use a dedicate if statement makes it more clear.

Conclusion

If nil handling has a very big meaning, not parallel to other case statement, it is better use a dedicate if.
